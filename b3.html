Section 1 · Configured Automated Controls

In Module B2 we established that preventive controls are the gold standard — they stop adverse events before they occur. But not all preventive controls are equally reliable. Control nature answers the question: how much does this control depend on the system versus a human being? The answer places every control on a descending scale of system reliance, from fully automated at the top through to fully manual at the bottom.

①Highest
Configured Automated
The system enforces the control automatically on every transaction, every time. No human action required in normal operation.
OMR6 Three-way match tolerance
Highest RelianceTest: Config inspection + sample postings
②Medium
IT-Dependent Manual (ITDM)
The system generates data; a human must review that output and take action. Both elements must be evidenced.
FK04 Monthly vendor master change report review
Medium RelianceTest: Report accuracy + review evidence
③Lowest
Manual
No IT component at all. Entirely dependent on human behaviour, memory, and consistency. Highest failure risk; largest testing sample required.

Lowest RelianceTest: Physical artefacts of performance

1.1 · Configured Automated — Deep Dive

The source session's exact words: "this is your best bet — the system is automatically going to be set up to prevent something." The three-way match is the canonical example. The system automatically compares the purchase order (ME21N), goods receipt (MIGO), and vendor invoice (MIRO). If the discrepancy exceeds the configured tolerance in OMR6, the system blocks the invoice — automatically, consistently, on every single transaction, every time.

💡
Why This is the Highest Reliability Level
Configured automated controls do not depend on anyone remembering to perform a check. They do not weaken when staff are absent, busy, or under pressure. They enforce the same rule on the first transaction of the day and the last. This consistency is exactly why they attract the highest audit reliance rating — and why audit sample sizes for these controls are typically small.

1.2 · Ronnie's B2 Distinction — Applied to Testing

🔑 Extending Ronnie's B2 Insight
"An automated control could be based on system configuration, or that control could be inherent to the system — an out-of-the-box standard control. There is a distinction."
— Ronnie · Source Session AUD-20260602-WA0018

In B2, Ronnie flagged the distinction between configured and inherent automated controls. In B3, we make that distinction operationally concrete — because it changes your test procedure:

Configured Automated
Requires a Configuration Review
Enabled by deliberate setup. OMR6 three-way match tolerance. Test step: inspect the OMR6 settings and verify the tolerance parameters are set appropriately — not just confirm "three-way match is enabled."

A tolerance of 99% technically means the control is configured, but operationally it provides almost no protection.
Inherent Automated
Requires a "Not Disabled" Check
Active by default — no configuration required. MIRO duplicate invoice check (document number + vendor combination). Test step: confirm the standard SAP functionality has not been switched off in system settings.

Some clients disable standard checks to accommodate workarounds — that disablement is itself a control gap.

1.3 · Writing a Complete Test Procedure

The most common weakness in configured automated control testing is an incomplete test procedure. Here is what the difference looks like in practice:

❌ Insufficient — Not Acceptable as Audit Evidence
"Confirm three-way match is enabled in the system."

This evidences existence, not effectiveness. It says nothing about whether the tolerance is appropriate, whether the configuration has been overridden, or whether the system actually blocked any discrepant invoices during the test period.
✅ Complete — Three-Component Test
1. Configuration inspection: Navigate to OMR6; confirm tolerance key is set to ≤3% or per approved policy.

2. Parameter verification: Confirm no user-specific or vendor-specific exceptions override the standard tolerance.

3. Sample test: Select 25 MIRO postings from the test period; confirm the system blocked invoices where discrepancy exceeded tolerance and generated a blocking reason code.

Section 2 · IT-Dependent Manual (ITDM) Controls

Where we cannot have everything be configured and automated, we leverage the IT-Dependent Manual (ITDM) control. The source session defined it precisely: it does require human interaction, but you are at least leveraging the system to a certain extent. The canonical example is a monthly review: generate a report from the system, then require somebody to review that report, identify the discrepancies, and take action.

💻
System Component
IT-Generated Data
SAP generates the report, log, or exception list automatically. Examples: monthly vendor master change report (FK04), GRC SoD conflict list, Financial Closing Cockpit variance output, payment run exception report (F110 log).
+
👤
Human Component
Manual Review & Action
A competent person reviews the output, investigates exceptions, and takes documented corrective action. The control's effectiveness depends entirely on the quality and consistency of this human element.
⚠️
The Core Reliance Risk
An ITDM is only as strong as the quality and consistency of the human review. The system component is reliable — it generates the report on schedule. The human component is where control quality degrades: reviewers who rush, miss exceptions, or fail to document their follow-up. This is why ITDM controls attract medium reliance ratings, not high.

2.1 · Testing an ITDM — Both Elements Required

Testing an ITDM requires evidence of both elements. This is the most common source of ITDM audit deficiencies — practitioners test one element and miss the other.

01
Coverage Check
Does the report cover all relevant transactions for the test period? Verify the date range, entity scope, and transaction type filters are correctly configured.
02
Data Source Verification
Is the report pulling from the correct system table or data source? Trace the report logic to the underlying SAP data.
03
Accuracy Sample
Select 5–10 transactions from the underlying data and agree them to the report output. Confirm no exceptions are missing from the population.
01
Evidence of Review
Is there documented evidence the reviewer examined the report — a sign-off, date stamp, or email approval? Oral confirmation ("we always review it") is not audit evidence.
02
Exception Follow-Up
For each exception flagged in the period: is there documented follow-up and corrective action? A signed-off report with no exception follow-up is evidence that a review occurred — not that it was effective.
03
Reviewer Competence
Was the reviewer competent and appropriately independent? Check the reviewer's role against the process being reviewed — a reviewer approving their own transactions is not independent.
⚠️
The Most Common ITDM Deficiency
A signed-off report with no exception follow-up is evidence that a review occurred — not that it was effective. This distinction drives some of the most common ITDM audit findings in SOX engagements. Control owners who sign reports without investigating or documenting exception resolution create a paper trail that looks like a working control but evidences nothing about effectiveness.

Section 3 · Manual Controls & The Uplift Imperative

At the bottom of the scale sits the fully manual control — no IT component at all. The source session is direct: these are going to be the least effective, and you want to leverage the more configured and ITDM controls as much as we can. But manual controls are a reality on every engagement, particularly in current-state assessments of legacy environments.

3.1 · Manual Control Nature

A manual control depends entirely on human behaviour — on memory, consistency, diligence, and the integrity of the individual performing it. There is no system prompt, no automated flag, no configuration enforcement. If the person forgets, the control fails. If the person rushes, the control is weakened.

Failure ModeMechanismAudit Implication
Person forgetsNo system prompt or reminder — control simply does not occurNo evidence of performance for the period; control gap
Person rushesReview completed superficially — exceptions not investigatedEvidence of review exists but effectiveness cannot be confirmed
Staff turnoverControl knowledge not documented; successor unawareInvisible gap — may persist across multiple audit periods
Evidence manipulationPaper easier to falsify than system-generated logsAuditor must consider documentary integrity, not just existence

3.2 · Testing Manual Controls

Testing manual controls requires physical artefacts of the control having been executed — signed forms, dated approval stamps, email confirmations, documented inspection records. The physical record must exist, be complete, and cover the period under review.

⚠️
Oral Confirmation is Not Evidence
"We always do it this way" is not audit evidence for a manual control. The physical record of the control having been performed is evidence. Auditors must also consider documentary integrity: manual evidence is easier to falsify than system-generated logs, which is why manual controls attract the largest sample sizes and, where evidence manipulation is a concern, may require corroborating evidence from independent sources.

3.3 · The Uplift Imperative — From Manual to Automated

On every S/4HANA implementation engagement, one of the highest-value contributions you can make is to identify manual controls in the current state and recommend their conversion to configured automated or ITDM controls in the future state. The source session captures this philosophy exactly: "if you do this this way today, maybe tomorrow you can do it better."

Current State
Manual
Paper goods receipt approval form. Manager signs physically. Filed in binder. Loses consistency with staff turnover. Large sample size required annually.
Interim State
ITDM
SAP generates weekly goods receipt exception report. Manager reviews and signs off in the system. Better than manual — but still dependent on review quality.
Future State
Configured Automated
SAP tolerance configured in OMR6 or goods receipt quantity tolerance. System blocks postings outside tolerance automatically. Small sample size. No human dependency.
✅
The Hierarchy Made Operational
Moving a manual paper check to a configured OMR6 tolerance rule is not just a compliance improvement — it eliminates a human failure point, reduces the annual cost of testing, and raises the reliability of the entire controls framework. That is a direct contribution to the client's long-term controls maturity. This is the practitioner's value-add during an S/4HANA transformation.

Key Definitions
TermControl Nature
The degree to which a control depends on the IT system versus a human being. Falls on a descending scale from configured automated (full system reliance, highest reliability) through ITDM (shared reliance) to manual (zero system reliance, lowest reliability). Control nature determines both the expected reliability and the required test approach.
TermConfigured Automated Control
A control that the IT system enforces automatically on every transaction, without human action. Requires deliberate configuration setup (e.g. OMR6 tolerance keys) or is inherent to standard SAP functionality (e.g. MIRO duplicate invoice check). Testing must distinguish: configured controls require a configuration review; inherent controls require confirmation they have not been disabled.
TermIT-Dependent Manual (ITDM)
A control that leverages IT to generate data (report, log, exception list) but requires a human to review and act on that output. Testing requires two-element evidence: (1) the system output is complete and accurate, AND (2) a competent person reviewed it and took documented action on exceptions. Absence of exception follow-up means the control's effectiveness is not evidenced.
TermManual Control
A control with no IT component — entirely dependent on human behaviour. Lowest reliance; requires the largest sample sizes in testing. Physical artefacts of performance (signed forms, dated stamps, email confirmations) are required as evidence. Oral confirmation is not acceptable audit evidence. The uplift imperative: convert manual controls to ITDM or automated wherever the system allows during S/4HANA transformation.
TermUplift Imperative
The practitioner's design obligation during S/4HANA implementation to recommend and design controls at the highest feasible level of automation. A manual control converted to a configured automated control eliminates a human failure point, reduces annual testing cost, and raises the overall reliability of the controls framework. This is a direct contribution to long-term controls maturity — not just a compliance improvement.
