Section 1 · What Is an FD and the Risk Filter

A functional design document is a programme artefact produced by the system implementer. It describes how a specific function will work in the future-state system — an interface between two systems, a workflow that automates an approval, a custom report, or a configuration rationale. The source session is direct: the idea is not to read word for word, but to understand what specific types of details you need to draft an appropriate control.

The One Question Before You Open the Document

Before reading a single page of an FD, answer this question: What risk am I trying to control? This is your filter. Without it you will read everything and extract nothing. Define the risk — financial, operational, or regulatory — then assess whether this FD is relevant. If the answer is yes, proceed to parameter extraction.

PetroNova S.A. Scenario — FD-MM-INT-027
This module uses FD-MM-INT-027 — a purchase requisition interface from VANTAGE ERP into SAP S/4HANA MM — as the running example. The risk: if PRs are not transferred completely and accurately, PetroNova cannot raise purchase orders in SAP. That is an operational risk (procurement stops) and a financial risk (purchases made off-system create unrecorded liabilities). The FD is relevant. Proceed to extraction.

Section 2 · The Four Extraction Parameters

Once you have confirmed the FD is relevant to your risk, move into extraction mode. For an interface FD — the most common type in a Materials Management or Finance transformation — four parameters generate your control design.

1
Business Purpose
"What is this interface supposed to do — and what is the business consequence if it fails?"
Business purpose connects the technical specification to the financial or operational risk. A procurement interface that stops working means purchase orders cannot be raised. A financial posting interface that fails means the general ledger is incomplete.
FD-MM-INT-027 → Section 2: Transfers approved PRs from VANTAGE ERP to SAP S/4HANA MM. Failure: no POs raised → procurement stops or goes off-system.
2
Direction
"Is this unidirectional or bidirectional?"
Direction determines where failure must be detected. For a unidirectional interface, all failure detection must happen at the receiving system — the sending system has no visibility of what happened after it transmitted. The monitoring control must sit at the receiver.
FD-MM-INT-027 → Section 3.2: UNIDIRECTIONAL — VANTAGE sends, SAP receives. VANTAGE receives no confirmation. All detection on the SAP side.
3
Frequency
"How often does the interface run?"
Frequency sets the maximum exposure window — the period during which undetected errors can accumulate. A daily monitoring report is not sufficient for an interface that runs every hour. The control design must match the frequency.
FD-MM-INT-027 → Section 3.3: Every 4 hours during business hours (normal operations). Exposure window = 4 hours per cycle.
4
Error Handling ⚠
"How does the system detect, alert, and resolve transfer failures?"
This is where most interface controls live. You need: automatic error detection, an alert mechanism, a defined resolution procedure with a timeline, and a reconciliation to confirm the data was recovered. If any of these is missing, that is a control gap.
FD-MM-INT-027 → Section 5: Hard Errors → /PNS/INT_ERR → email alert within 15 min → 4-hour resolution SLA → re-submission log → daily sign-off report.

Section 3 · Deriving the Control

With all four parameters extracted, the control design follows directly. Using the PetroNova S.A. FD-MM-INT-027 scenario:

RACM Control Output — FIN-P2P-028
Control Reference
FIN-P2P-028
Control Type
Detective
Control Nature
ITDM — IT-Dependent Manual
Risk
Incomplete/inaccurate PR transfer → no POs in SAP → off-system procurement → unrecorded liabilities
Control Description
The P2P Interface Monitor reviews the daily SAP monitoring report (/PNS/INT_RPT_DAILY) by 09:00 WAT each business day. For each Hard Error flagged, the Monitor investigates the root cause, resolves it, and re-submits the affected records within 4 hours of notification. The signed-off monitoring report and re-submission log are retained as evidence.

Why This Is an ITDM Control

The system component: SAP automatically generates the monitoring report and populates the error log. The human component: the P2P Interface Monitor reviews the report, investigates exceptions, documents corrective action, and signs off. Both elements must be present and evidenced — a report that is generated but not reviewed is evidence that a control exists, not that it is effective. This is the B3 ITDM pattern applied to an interface control scenario.

Test Procedure — Two-Element ITDM Pattern

01
Coverage Check
Confirm the daily monitoring report covers all batch cycles for the test period. Verify the report is generated from the correct SAP table (/PNS/INT_ERR and /PNS/INT_WARN).
02
Accuracy Sample
Select 3 batch cycles from the test period. For each, agree the Hard Error count in the report to the records in /PNS/INT_ERR. Confirm there are no gaps — every hard error in the table should appear in the report.
03
Reconciliation Verification
Confirm the end-of-day PR count reconciliation was performed — total approved PRs in VANTAGE matches total PR records received by SAP for the same day.
01
Review Evidence
Is there a signed-off monitoring report for each business day in the test period? Confirm sign-off is by the P2P Process Lead (not a subordinate or IT). Confirm sign-off occurred by 09:00 WAT the following business day per the SLA.
02
Exception Follow-Up
For each Hard Error flagged in the period: is there a re-submission record in /PNS/INT_RESUBMIT with a timestamp within 4 hours of the original alert? Is there a documented root cause and resolution sign-off?
03
Effectiveness Assessment
A monitoring report that is signed off with no exception investigation documented — despite hard errors in the log — is evidence that a review occurred, not that it was effective. Flag as ITDM deficiency if exception follow-up is absent.

The Four Extraction Parameters — Quick Reference
Parameter 01
Business Purpose
FD Section 2. What data transfers, between which systems, and what is the business consequence if the transfer fails? This connects the technical spec to the financial or operational risk in the RACM.
Parameter 02
Direction
FD Section 3. Unidirectional or bidirectional? Unidirectional: all failure detection must sit at the receiving system. The sending system has no visibility after transmission.
Parameter 03
Frequency
FD Section 3. Sets the exposure window — how long errors accumulate before the next monitoring cycle. Control monitoring frequency must match or exceed the transfer frequency.
Parameter 04 ⚠
Error Handling
FD Section 5. Controls-Critical. Must cover: automatic detection, alert mechanism, resolution SLA, reconciliation/sign-off evidence. Any gap = a control gap.
Key Definitions
TermFunctional Design Document (FD)
A programme artefact produced by the system implementer describing how a specific function will work in the future-state system. Types: interface FD, workflow FD, custom report FD, configuration rationale. Controls teams read FDs to identify risk-relevant design decisions and extract control parameters — not to read them word for word.
ConceptRisk Filter
The one question that must be answered before opening an FD: "What risk am I trying to control?" This is the practitioner's filter — it determines whether the FD is relevant at all, and it shapes which sections of the FD to prioritise. Defined as financial, operational, or regulatory.
ConceptExposure Window
The period during which undetected errors can accumulate between monitoring cycles. Set by the interface frequency. If an interface runs every 4 hours and errors are only reviewed once daily, the exposure window is up to 20 hours. Control design must close this gap — monitoring frequency must match or exceed transfer frequency.
ConceptControls-Critical Section
The section of an FD that directly generates control parameters — typically the error handling and monitoring section (Section 5 in FD-MM-INT-027). Practitioners should navigate directly to this section after confirming relevance and direction. It describes what happens when things go wrong — and that is where the control lives.
ControlFIN-P2P-028 — Interface Monitoring Review
The ITDM control derived from FD-MM-INT-027. System component: SAP generates the daily monitoring report from /PNS/INT_RPT_DAILY. Human component: P2P Interface Monitor reviews, investigates exceptions, and signs off. Detective in nature (identifies errors after they occur). Both elements must be evidenced for the control to be testable under the Module B3 ITDM standard.
