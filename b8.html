Testing Phase — Three Decisions

Testing is where controls earn their credibility — or expose their gaps. Three decisions determine how effective your testing phase will be: what counts as sufficient evidence, how to scope SIT versus UAT, and how to anchor your coordination through the testing captain.

Decision 1 · What Counts as Sufficient Evidence?

Testing is the best and last chance to see if the controls will actually work as expected. The evidence standard determines whether your test result is defensible — to your client, to an external auditor, and to your own quality review. Each test category has a different evidence standard.

🏭 Field Example — Major Energy Sector Engagement
On a major energy sector engagement, the three-way match configuration test involved navigating to SPRO → the tolerance key path → confirming the tolerance limits were set. The design specified a 3% threshold. What the system showed: 2% for both quantity and price variance. More restrictive than designed — test result: Pass. But the value-add came next: reviewing all company codes, not just one. Most showed 2–3%. One showed 10%. That exception needed a policy justification or a correction. A practitioner who checked only the first company code would have missed it.

Evidence Standard by Test Category

Insufficient: A screenshot showing "three-way match active" or "workflow enabled." This proves existence — not appropriateness.

Sufficient: Three components required:

01
Navigation path
Screenshot showing the exact SPRO or transaction path taken to reach the configuration setting. Not just the result screen — the path to it.
02
Value confirmation
Screenshot showing the specific tolerance or threshold value — e.g. 2% for quantity variance and 2% for price variance. Compared against the design threshold in the RACM.
03
Company code cross-check
All relevant company codes reviewed. Any exception to the standard value is documented. If one company code has 10% where all others have 3%, that exception requires a policy justification or correction before go-live.

The rule: Every transactional test requires a positive scenario AND a negative scenario. A test plan with only positive scenarios does not prove a control blocks what it should block.

✅ POSITIVE SCENARIO
Compliant transaction processes correctly. For three-way match: invoice within 2% tolerance is posted — no block fired.
Purpose: confirms no false positives.
❌ NEGATIVE SCENARIO
Non-compliant transaction is blocked. For three-way match: invoice exceeding 2% tolerance — system blocks posting and generates blocking reason code.
Purpose: confirms the control actually blocks what it should.
⚠️
The Most Common Transactional Testing Deficiency
Test plans that contain only positive scenarios. When this happens: push back. "We do not want to just see the happy path. We want to see what happens when it works and what happens when it does not work as expected."

✅ Sufficient — for Transactional Testing
Both scenarios are present: a positive scenario (compliant transaction processes without a block) and a negative scenario (non-compliant transaction triggers the system block with a blocking reason code). This is sufficient evidence that the control is operating as designed for transactional testing purposes.
Remember: This covers transactional testing only. A complete test package for a configured automated control also requires the configuration evidence (Screenshot B + company code cross-check) as a separate test step.

Decision 2 · SIT vs UAT Scope

Which controls belong in System Integration Testing, and which belong in User Acceptance Testing? The split is based on control nature — not risk rating, not design ownership.

⚙️ SIT — System Integration Testing

Rule: Configuration and automated controls — those that operate without human intervention.

The system itself is being validated. Technical testers are executing functional test cases. Controls requirements are integrated into the programme test scripts — testers execute and produce the required evidence in one pass.

SIT EXAMPLES
Three-way match tolerance (OMR6) · Vendor master workflow configuration · Duplicate invoice block · Payment run SoD restriction
👤 UAT — User Acceptance Testing

Rule: ITDM and manual controls — those that require human review and action.

Business users are in the system. They can demonstrate they can perform the required review steps — generate the report, investigate exceptions, sign off. Both the IT component and the human component are evidenced.

UAT EXAMPLES
PR Interface Monitoring Report Review · Vendor Change Report Review · Period-End Close Checklist · Emergency PO Authorisation Review
💡
Why Automated Controls Don't Need to Be Re-Run in UAT
Configuration controls validated in SIT do not change between SIT and UAT — the system configuration is the same. Re-running them in UAT consumes business user time and testing programme capacity without adding audit value. The efficiency gain from the SIT/UAT split is real — on a programme with 300 controls, it can eliminate dozens of unnecessary UAT test cycles.

Decision 3 · The Coordination Anchor — Testing Captain

How do you maintain evidence quality across a testing programme with dozens of testers, hundreds of test cases, and multiple coordination layers? The testing captain is the answer.

A testing captain is the coordinator for a group of testers in the programme's testing structure. They are not necessarily senior — they may be a test coordinator or team lead from the system integrator's testing organisation. What they provide is a single point of contact for evidence requests, rework, and re-execution.

Without the testing captain model: the controls team chases each individual tester directly. Each tester may interpret evidence requirements differently. Re-test requests are managed across hundreds of separate email threads. Evidence quality is inconsistent.

With the testing captain model: evidence requests go to one person. The captain enforces consistent standards across their tester group. Re-test requests are coordinated through one channel. The controls team's time is spent reviewing evidence, not chasing individuals.

01
Map controls to test cases
Before testing begins, review each programme test case and identify which controls it exercises. Add the RACM control reference to the test case metadata.
02
Add controls requirements to test scripts
For each mapped test case, add the specific controls testing requirements as additional steps. When a tester opens the test case, they see both the programme requirement and the controls requirement in the same document.
03
Specify evidence requirements upfront
Publish the evidence requirements before testing starts — exactly which screenshots, system logs, or outputs are needed for each control test. Agree with the testing captain before the first test is executed.
04
Define the re-test mechanism
Define in advance: who decides if evidence is insufficient, who communicates the re-test request, and what the SLA is. The testing captain is the communication channel. Setting this up before testing starts eliminates the friction of chasing re-executions mid-programme.
✅
The Practitioner's Payoff
Setting this up takes effort in Design and Build. It pays back in the Testing phase every time you do not have to chase a re-execution.

Key Definitions
TermPositive / Negative Test Scenario
Positive scenario: a compliant transaction that should process without a block — confirms no false positives. Negative scenario: a non-compliant transaction that should be blocked — confirms the control fires when required. Both are required for transactional testing. A test plan with only positive scenarios does not prove a control is operating as designed.
TermCompany Code Cross-Check
The practice of reviewing configuration settings across all relevant SAP company codes, not just the primary one. Tolerance keys, workflow thresholds, and other configuration settings may be set at the company code level. An exception — one company code at 10% when all others are at 3% — must be investigated and either justified by policy or corrected before go-live.
TermSIT vs UAT Scope Split
The decision about which controls to test in System Integration Testing versus User Acceptance Testing. Based on control nature: automated/configuration controls go to SIT (system being validated technically); ITDM and manual controls go to UAT (business users demonstrating the human element). The split distributes workload and prevents unnecessary re-testing.
TermTesting Captain
The coordinator for a group of testers in the programme's testing structure. The controls team's single point of contact for evidence requests, rework instructions, and re-test coordination within that tester group. The testing captain model eliminates the need to chase individual testers directly and enforces consistent evidence standards across the group.
ConceptIntegration Before Testing Starts
The practice of mapping controls to programme test cases, adding controls testing requirements to test scripts, specifying evidence requirements, and defining the re-test mechanism — all before the first test is executed. This work is done in Design and Build. It eliminates the most common source of testing phase friction: insufficient evidence discovered mid-testing with no pre-agreed process for rework.
